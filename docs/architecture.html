Boundary Rule

Frontend apps (studio, desktop) MUST NOT import directly from packages. All data flows through the API via HTTP. This ensures clean separation and allows the API to enforce validation, authentication, and business logic.

Book Directory Structure

Each book is fully self-contained — zip it, share it, everything travels together.

my-book/
├── book.json # Book metadata & config
├── source/
│ └── input.pdf # Original uploaded PDF
├── entities/
│ ├── chapters/
│ │ ├── ch-001.v1.json # Version 1
│ │ ├── ch-001.v2.json # Version 2 (never overwrite!)
│ │ └── ch-001.v3.json # Latest version
│ ├── characters/
│ └── scenes/
├── .cache/ # LLM call cache
│ ├── a3f2e1...json # hash(model + prompt + params) → response
│ └── b7c4d9...json
├── .logs/ # Full LLM call transparency
│ ├── 2025-02-04T10-30.json # Every prompt & response logged
│ └── cost-summary.json # Token usage & cost tracking
└── output/
    └── bundle.zip # Final packaged output

Entity Versioning

Entities are NEVER overwritten. Each modification creates a new version file (v1, v2, v3...) with an incremented version number. This enables full rollback capability and audit trails. The inputVersions field tracks which versions of upstream entities were used to generate each version.

LLM Caching

Every LLM call is cached by hashing all ordered inputs (model, prompt, parameters). If a pipeline step is re-run with identical parameters, the cached response is returned instantly. Cache lives inside the book directory so it travels with the book.

Technology Decisions

Click any card to see detailed reasoning.

pnpm Package Manager
Monorepo-first package manager with efficient disk usage
Over: npm, yarn
  • Content-addressable storage via symlinks — saves significant disk space
  • Strict dependency resolution prevents phantom dependencies
  • First-class workspace support with filtering and topology-aware commands
  • Faster installs than npm/yarn in monorepo scenarios
  • Built-in support for workspace:* protocol
Hono HTTP Server
Lightweight, fast web framework with native TypeScript
Over: Express, Fastify
  • ~14KB — significantly lighter than Express
  • Built-in TypeScript support with typed routes and middleware
  • Faster request handling benchmarks vs Express
  • Edge-ready (works on Cloudflare Workers, Deno, Bun)
  • Middleware ecosystem: CORS, body parsing, error handling built-in
  • Perfect for embedding in Tauri sidecar (minimal footprint)
React + Vite Frontend
SPA with lightning-fast dev server — desktop-first, no SSR needed
Over: Next.js, Create React App
Why NOT Next.js:
  • ADT Studio is a desktop app in Tauri — Tauri serves static files from disk, not from a Node.js server
  • Next.js brings SSR, server components, API routes — none apply to a local desktop SPA
  • Next.js output (.next/) expects a Node.js runtime to serve it; Vite outputs a static dist/
  • Next.js in a monorepo requires transpilePackages, serverComponentsExternalPackages config
  • No SEO needed (desktop app), no initial page load from remote server
Why Vite:
  • Near-instant HMR via native ES modules
  • Simple config, minimal build output
  • First-class TypeScript and JSX support
  • Easy monorepo setup with workspace packages
Tauri Desktop
Lightweight, secure native desktop wrapper
Over: Electron
  • ~10x smaller bundle than Electron (uses system webview)
  • Rust backend for performance-critical operations
  • Better security model with allowlist-based permissions
  • Lower memory footprint — no bundled Chromium
  • Cross-platform: Windows, macOS, Linux
  • Can run the Hono API as a sidecar process
Tailwind CSS Styling
Utility-first CSS framework for consistent design
Over: CSS Modules, styled-components
  • Eliminates CSS bloat — unused styles are purged
  • Consistent design system with spacing, color, and typography scales
  • No context switching between CSS and JSX files
  • Excellent IDE support with IntelliSense
  • Conditional classes via clsx — clean, readable
  • Aligns with "minimize dependencies" principle — no CSS-in-JS runtime
Zod Validation
Schema-first validation with TypeScript inference
Over: io-ts, Yup, Joi
  • Runtime + compile-time validation from a single schema
  • z.infer<typeof Schema> — no duplicate type definitions
  • Excellent TypeScript integration and DX
  • Schema composition, transforms, and refinements
  • .safeParse() for non-throwing validation at API boundaries
  • Smaller and faster than Yup/Joi, better TS support than io-ts
Vitest Testing
Vite-native test runner with Jest-compatible API
Over: Jest
  • Native ESM support without configuration hacks
  • Reuses Vite's transform pipeline — consistent with dev
  • Faster execution via worker threads
  • Jest-compatible API (describe, it, expect, vi.mock)
  • Built-in coverage via c8/v8
  • Watch mode with instant re-runs
node-sqlite3-wasm Database
Pure WASM SQLite — no native compilation, cross-platform
Over: better-sqlite3, sql.js, Prisma
  • Pure WASM — no node-gyp, no C++ compiler, no platform-specific binaries
  • File system persistence via Node.js VFS (unlike sql.js which is in-memory only)
  • Synchronous API — same DX as better-sqlite3
  • Based on SQLite 3.51.1 — current and maintained
  • Cross-platform by default — works on any OS without recompilation
  • File-based = portable (lives inside book directory)
  • Aligns with "pure JS/TS over native" and "book-level storage" principles
TanStack (Full Ecosystem) Frontend Core
Unified, type-safe ecosystem: Router + Query + Table + Form
Over: React Router, SWR, react-hook-form, react-table
Router — Type-safe routes, search params as state, integrated loaders. React Router v7 only has type safety in framework mode, not SPA mode.

Query — Caching, stale-while-revalidate, auto-refetch, mutations with optimistic updates. Replaces manual useEffect + polling patterns.

Table — Headless sorting, filtering, pagination, virtualization. We provide Tailwind-styled UI.

Form — Type-safe forms with native Zod adapter. Same schemas used at API boundary validate forms at the UI layer.

  • One team, one philosophy — libraries designed to work together
  • Type safety flows from Router → Query → Table → Form
  • No glue code between different library paradigms
  • Millions of weekly downloads, actively maintained by Tanner Linsley
clsx Utility
Tiny conditional className helper
Over: classnames
  • ~228 bytes — smaller than classnames
  • Same API — drop-in replacement
  • Better tree-shaking
  • Perfect for conditional Tailwind classes

6 Core Design Principles

These are non-negotiable and must guide every architectural decision.

📁
Book-Level Storage
All book data lives in a single directory. Zip it, share it — everything travels together. Never store book data outside its directory.
🔄
Entity Versioning
Never overwrite entities. Always create new versions (v1, v2, v3...). Users can roll back to any previous version at any time.
⚡
LLM-Level Caching
Cache at the LLM call level by hashing inputs. Pipeline reruns are instant when parameters haven't changed. Cache lives in the book.
🔍
Maximum Transparency
Every LLM call, prompt, and response is inspectable. No black boxes. Users see exactly what the AI was asked and what it answered.
🪶
Minimize Dependencies
Avoid new deps when possible. Flat files over databases when sufficient. In-memory queues over external services. Keep it simple.
🧩
Pure JS/TS Over Native
Always prefer pure JS/WASM libraries over native C/C++ bindings. No node-gyp, no platform-specific builds. Cross-platform by default.

Security Architecture

Frontend Boundary
✓ API keys sent via X-OpenAI-Key header per request
✗ Keys never stored in localStorage or client bundle
✗ Keys never sent in URL parameters
✓ All data access goes through API — no direct package imports
API Validation Layer (Hono)
✓ ALL input validated with Zod schemas via .safeParse()
✓ API key presence checked via requireOpenAIKey()
✓ Path traversal prevention: path.resolve() + startsWith check
✓ SQL: parameterized queries only (db.prepare("...?").get(id))
✓ CORS: explicit origin allowlist, no wildcards
✓ HTTPException for structured error responses (400, 401, 404)
Storage & Processing
✓ API keys: environment variables only, never logged or persisted
✓ Atomic read-modify-write with file locking
✓ XSS prevention: React auto-escaping + DOMPurify for HTML
✓ No dangerouslySetInnerHTML without sanitization
✓ Error messages never expose API keys or internal details
